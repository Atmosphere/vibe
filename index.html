wes

Web Event Source.

The wes (Web Event Source) is an abstraction layer in the form of event source for various event-driven asynchronous web client and server that runs on the JVM allowing the end user to choose the desired library.

An application written with wes can run on any supported server and client, on Java.

Write once, run anywhere

Servlet specification had been designed to bring this paradigm to web applications, but it could not have lasted long. Because of the lack of support of asynchronous IO, uncomfortable API and so on, so many application framework and IO framework arose to hide or drop to Servlet. That's why it comes as not easy to refer to some implementation written in Java as Java implementation. The author have to specify supported frameworks and do more work to support others.

The wes has been designed to run your application on as many as possible providing event-driven API, just as jQuery does on browser.

In fact it has started to help write true Java implementation of Portal, the realtime web application framework.

That being said, wes is not a silver bullet.

Write once, debug everywhere

As you already know, Run anywhere makes inevitable Debug everywhere. Even though wes is guaranteed to work with a framework you use, it doesn't ensure your application will work in production as expected. You should write integrated test suite for your application.

Full stack frameworks are from Mars, IO frameworks are from Venus

When it comes to interface to handle HTTP and WebSocket, full stack frameworks such as Play tend to have very simplified API but IO frameworks such as Netty tend to have very delicate API. Therefore, interfaces provided by wes are limited to their intersection and may be less rich than you expect.
